    What is SpecKit?

    intro overview

    SpecKit is an open-source framework that enables Spec-Driven Development (SDD) by transforming rich, natural language specifications into executable workflows.

    Key Features

    • Leverages advanced AI coding assistants to interpret specifications
    • Clarifies ambiguities through structured Q&A
    • Plans implementations and generates code automatically
    • Maintains traceability and version control throughout
    Core Principle
    Specifications become first-class citizens in development, not just documentation that gets discarded after coding begins.

    Problems Solved

    challenges, benefits

    Traditional Development Challenges

    • üìù Specifications often become outdated after coding begins
    • üéØ Gap between "what we want" and "what we build"
    • üé≤ Vibe-coding: jumping directly to implementation without clear structure
    • üí∏ Technical debt from unclear requirements
    • üó£Ô∏è Miscommunication between stakeholders and developers
    • üìä Inconsistent quality and completeness of requirements

    AI Development Challenges

    • ‚ö° Inefficient use of AI coding assistants without structured guidance
    • ü§ù Limited collaboration between humans and AI in development

    The Paradigm Shift

    ‚ùå Old Way: Specifications ‚Üí Discard ‚Üí Code ‚Üí Ship

    ‚úÖ New Way: Specifications ‚Üí Executable ‚Üí Implementation ‚Üí Ship

    Spec-Driven Development

    philosophy methodology

    Spec-Driven Development is a structured process that emphasizes three core principles:

    1. Intent-Driven Development

    • Define the "what" and "why" before the "how"
    • Focus on product scenarios, not technical implementation
    • Let AI interpret specifications and generate code

    2. Rich Specification Creation

    • Use guardrails and organizational principles
    • Multi-step refinement process
    • Not one-shot prompt engineering

    3. AI-Powered Execution

    • Heavy reliance on advanced AI model capabilities
    • Specifications become executable through AI agents
    • Technology-independent approach

    Key Benefits

    • ‚ö° Faster Development: Reduce time from idea to implementation
    • ‚úÖ Higher Quality: Structured approach ensures completeness
    • üí¨ Better Communication: Clear specifications for all stakeholders
    • üîß Technology Independence: Not tied to specific frameworks or languages
    • üìö Maintainability: Living documentation that stays current

    Development Phases

    greenfield, brownfield

    SpecKit supports three distinct development scenarios:

    1. 0-to-1 Development (Greenfield)

    Generate applications from scratch with high-level requirements
    • Start with high-level requirements
    • Build production-ready applications
    • Example: Building a new task management system

    2. Iterative Enhancement (Brownfield)

    Add features to existing applications and modernize legacy systems
    • Add features to existing applications
    • Modernize legacy systems
    • Adapt and evolve processes
    • Example: Adding real-time collaboration to existing app

    3. Creative Exploration

    Explore diverse solutions in parallel with different technology stacks
    • Explore diverse solutions in parallel
    • Support multiple technology stacks
    • Experiment with different UX patterns
    • Example: Testing React vs Vue vs Vanilla JS implementations

    Slash Commands

    commands, usage

    After running specify init, your AI coding agent will have access to 8 slash commands:

    Core Commands

    Command Description
    /speckit.constitution Create or update project governing principles and development guidelines
    /speckit.specify Define what you want to build (requirements and user stories)
    /speckit.plan Create technical implementation plans with your chosen tech stack
    /speckit.tasks Generate actionable task lists for implementation
    /speckit.implement Execute all tasks to build the feature according to the plan

    Optional Commands

    Command Description
    /speckit.clarify Clarify underspecified areas (recommended before /speckit.plan)
    /speckit.analyze Cross-artifact consistency & coverage analysis
    /speckit.checklist Generate custom quality checklists that validate requirements

    SpecKit Commands

    SpecKit provides a structured approach to software development through seven key commands:

    1. constitute - Define the initial project concept and goals
    2. specify - Create detailed specifications and requirements
    3. clarify - Ask questions and resolve ambiguities
    4. plan - Generate the overall project plan and architecture
    5. tasks - Break down the plan into actionable tasks
    6. analyze - Evaluate the plan for feasibility and completeness
    7. implement - Execute the tasks and build the solution

    Step 1: CONSTITUTE - Define the Project

    Command: /speckit.constitution

    Purpose: Establish the foundational concept and high-level goals for the project.

    Guidelines:

    • Focus on the core idea and objectives
    • Highlight key features and benefits
    • Define the target audience and success criteria
    • Optionally provide Technology Choices at this stage
    • Prepare for detailed specification in the next phase
    • Output a structured constitution document

    Input:

    /speckit.constitution Create principles focused on - 1. API-First Architecture - 2. Cloud-Native Design - 3. Test-Driven Development - 4. Security by Design - 5. Observability & Monitoring and Technology Choices

    SpecKit Output:

    In todoapp

.specify/memory/constitution.md

    Step 2: SPECIFY - Create Detailed Specifications

    Command: /speckit.specify

    Purpose: Transform the constitution into detailed, actionable specifications covering functional requirements, technical architecture, and constraints.

    Guidelines:

    • Break down high-level goals into specific features and requirements
    • Define data models, API endpoints, and interactions
    • Include non-functional requirements (performance, security, scalability)
    • Document constraints and assumptions
    • Prepare for clarification of ambiguities in the next phase

    Input:

    /speckit.specify Build a feature for creating and managing todo items

    Optional additions:

    • Functional requirements for task management (CRUD operations, filtering, sorting)
    • Data models for TaskItem, CreateTaskDto, UpdateTaskDto, and TaskPriority enum with validation rules
    • API endpoint definitions (routes, methods, request/response schemas)
    • Non-functional requirements: performance, scalability, security, testability (unit tests with xUnit, Moq, FluentAssertions, >80% coverage)
    • Constraints and assumptions (single user for MVP, no authentication, partition key strategy)

    SpecKit Output:

    In ToDoApp Project

specs\001-todo-management\spec.md

    Step 3: CLARIFY - Resolve Ambiguities

    Command: /speckit.clarify

    Purpose: Identify unclear aspects and ask questions to ensure complete understanding before planning.

    Guidelines:

    • Review the specifications for ambiguities or missing information
    • Formulate specific questions to clarify requirements

    • Prepare for finalizing specifications in the next phase

    Input (Optional):

    NA - We need to clarify any questions before proceeding to planning.

    Step 4: PLAN - Create Overall Project Plan

    Command: /speckit.plan

    Enhancement Feature: Priority and Due Dates

    Let's walk through adding priority levels and due dates to our TODO items using the SpecKit workflow for existing projects.

    Step 1: Analyze Existing Codebase

    Command: /speckit.analyze

    Purpose: Understand the current architecture, data models, and API structure before making changes.

    Input:

    /speckit.analyze Analyze the existing TODO App codebase. Review the current TodoItem model, API endpoints, data access patterns, and Cosmos DB schema. Identify integration points for adding priority levels (High, Medium, Low) and due dates to TODO items. Consider backward compatibility and migration requirements for existing data.

    SpecKit Output:

    The analysis command reviews your existing codebase and generates a detailed report identifying:

    • Current data models and their properties
    • Existing API endpoints and their contracts
    • Integration points where changes need to be made
    • Migration considerations for backward compatibility
    Example Analysis Report: The TodoItem model currently has Id, Title, IsCompleted, and CreatedAt properties. To add priority and due dates, we'll need to modify TodoItem.cs, update TodoService operations, adjust API DTOs, and add validation logic‚Äîall while maintaining backward compatibility with existing data.

    Step 2: Create Enhancement Specification

    Command: /speckit.specify

    Input:

    /speckit.specify Create a specification for enhancing the TODO App with priority levels and due dates. The feature should allow users to assign a priority (High, Medium, Low) and optional due date to each TODO item. Ensure backward compatibility with existing TODO items that don't have these fields. Include sorting and filtering capabilities by priority and due date.

    SpecKit Output:

    The specification command generates a detailed requirements document that defines the enhancement:

    • Functional requirements for priority levels and due dates
    • API endpoint modifications and new query parameters
    • Data model changes with validation rules
    • Backward compatibility requirements
    • Success criteria for the enhancement
    Generated file:
    specs/002-priority-due-dates/spec.md

    This specification document includes:
    - Feature overview and objectives
    - Detailed functional requirements for priority (High/Medium/Low)
    - Optional due date field with validation
    - API enhancements (filtering, sorting, overdue detection)
    - Backward compatibility guarantees
    - Non-functional requirements (performance, migration)
    - Clear success criteria for implementation
    Key Output: A comprehensive specification document that serves as the single source of truth for the enhancement. This document will guide all subsequent phases (clarification, planning, implementation) and ensure alignment between stakeholders and developers.

    Step 3: Impact Analysis

    Command: /speckit.analyze

    Purpose: Perform impact analysis to identify all affected components, potential breaking changes, database migration requirements, API versioning needs, and testing requirements.

    Input:

    /speckit.analyze Focus on impact analysis: Identify all affected components for adding priority and due dates. Assess potential breaking changes, database migration requirements, API versioning needs, testing scope, and deployment strategy. Consider backward compatibility with existing clients and impact on existing TodoItems without these fields.

    SpecKit Output:

    The impact analysis command evaluates all affected components, potential breaking changes, and migration requirements:

    • Component impact assessment (models, services, controllers, DTOs, tests)
    • Breaking change analysis (risk level and mitigation strategies)
    • Migration strategy for existing data (Cosmos DB schema handling)
    • Testing requirements (regression, integration, performance)
    • Deployment strategy (blue-green deployment approach)
    • Rollback plan for safe recovery if issues arise
    Example Impact Analysis: Adding priority and due dates is LOW RISK because all changes are additive. Existing TodoItems without these fields will use default values (Medium priority, null due date). No schema migration required in Cosmos DB. Backward compatibility maintained with existing clients. Blue-green deployment ensures safe rollout with quick rollback capability.

    Step 4: Create Implementation Plan

    Command: /speckit.plan

    Input:

    /speckit.plan Create a detailed implementation plan for adding priority and due dates to the TODO App. Break down the work into phases, identify dependencies, estimate effort, and define acceptance criteria for each task. Include testing strategy and rollback procedures.

    SpecKit Output:

    The plan command generates a detailed, phased implementation approach with tasks, timelines, and dependencies:

    • Phase 1: Data Model Updates (2-3 hours) - TodoItem enhancements, DTOs, validation
    • Phase 2: Service Layer Updates (3-4 hours) - Filtering, sorting, query optimization
    • Phase 3: API Controller Updates (2-3 hours) - Endpoints, parameters, documentation
    • Phase 4: Testing (3-4 hours) - Unit, integration, and regression tests
    • Phase 5: Documentation (1-2 hours) - API docs, user guides, migration notes
    • Phase 6: Deployment (1 hour) - Staging and production rollout
    Total Estimated Effort: 12-17 hours with clear dependencies, acceptance criteria for each task, and risk mitigation strategies. The plan ensures backward compatibility, comprehensive testing, and safe deployment.

    Command: /speckit.implement

    Input:

Update existing code carefully to maintain backward compatibility. Write comprehensive tests to verify both new features and existing functionality. Document all changes and prepare deployment artifacts.

    SpecKit Upgrade

    maintenance updates

    Keep your SpecKit installation up-to-date to access new features, bug fixes, and updated slash commands. This guide covers upgrading both the CLI tool and project files.

    Quick Reference

    What to Upgrade Command Purpose
    CLI Tool Only uv tool install specify-cli --force --from git+https://github.com/github/spec-kit.git Get latest CLI features without touching project files
    Project Files specify init --here --force --ai <your-agent> Update slash commands, templates, and scripts
    Both Run CLI upgrade, then project update Recommended for major version updates

    Part 1: Upgrade the CLI Tool

    The CLI tool (specify) is separate from your project files. Upgrade it to get the latest features and bug fixes.

    If you installed with uv tool install

    uv tool install specify-cli --force --from git+https://github.com/github/spec-kit.git

    If you use one-shot uvx commands

    No upgrade needed‚Äîuvx always fetches the latest version. Just run your commands as normal:

    uvx --from git+https://github.com/github/spec-kit.git specify init --here --ai copilot

    Verify the upgrade

    specify check

    This shows installed tools and confirms the CLI is working.

    Part 2: Updating Project Files

    What gets updated?

    Running specify init --here --force will update:

    • ‚úÖ Slash command files (.claude/commands/, .github/prompts/, etc.)
    • ‚úÖ Script files (.specify/scripts/)
    • ‚úÖ Template files (.specify/templates/)
    • ‚úÖ Shared memory files (.specify/memory/) - ‚ö†Ô∏è See warnings below

    What stays safe?

    These files are never touched by the upgrade:

    • ‚úÖ Your specifications (specs/001-my-feature/spec.md, etc.) - CONFIRMED SAFE
    • ‚úÖ Your implementation plans (specs/001-my-feature/plan.md, tasks.md, etc.) - CONFIRMED SAFE
    • ‚úÖ Your source code - CONFIRMED SAFE
    • ‚úÖ Your git history - CONFIRMED SAFE

    The specs/ directory is completely excluded from template packages and will never be modified during upgrades.

    Update command

    Run this inside your project directory:

    specify init --here --force --ai <your-agent>

    Replace <your-agent> with your AI assistant (copilot, claude, cursor, etc.)

    Understanding the --force flag

    Without --force, the CLI warns you and asks for confirmation. With --force, it proceeds immediately.

    Important: Your specs/ directory is always safe. The --force flag only affects template files (commands, scripts, templates, memory). Your feature specifications, plans, and tasks in specs/ are never included in upgrade packages.

    ‚ö†Ô∏è Important Warnings

    1. Constitution file will be overwritten

    Known issue: specify init --here --force currently overwrites .specify/memory/constitution.md with the default template, erasing any customizations.

    Workaround:

    # 1. Back up your constitution before upgrading
cp .specify/memory/constitution.md .specify/memory/constitution-backup.md

# 2. Run the upgrade
specify init --here --force --ai copilot

# 3. Restore your customized constitution
mv .specify/memory/constitution-backup.md .specify/memory/constitution.md

    Or use git to restore it:

    # After upgrade, restore from git history
git restore .specify/memory/constitution.md

    2. Custom template modifications

    If you customized any templates in .specify/templates/, the upgrade will overwrite them. Back them up first:

    # Back up custom templates
cp -r .specify/templates .specify/templates-backup

# After upgrade, merge your changes back manually

    3. Duplicate commands warning

    If you previously initialized with a different agent (e.g., claude) and now run with a new agent (e.g., copilot), you may get duplicate command files in different directories. Clean up manually if needed:

    # Check for duplicates
ls .claude/commands/
ls .github/prompts/

# Remove old agent files if switching permanently
rm -rf .claude/  # If switching from Claude to Copilot

    Rolling Back an Upgrade

    If an upgrade causes issues, you can rollback to a previous version.

    Rollback CLI Tool

    # Install specific version
uv tool install specify-cli --force --from git+https://github.com/github/spec-kit.git@v0.0.21

# Or reinstall from a specific tag
uv tool install specify-cli --force --from git+https://github.com/github/spec-kit.git@release-0.0.21

    Rollback Project Files

    # Use git to restore previous version
git restore .github/ .specify/

# Or revert to a specific commit
git log --oneline -- .github/ .specify/
git checkout <commit-hash> -- .github/ .specify/
    üí° Tip: Always commit your changes before upgrading so you can easily rollback if needed.

    Version Compatibility

    SpecKit follows semantic versioning for major releases. The CLI and project files are designed to be compatible within the same major version.

    Best practice: Keep both CLI and project files in sync by upgrading both together during major version changes.

    Next Steps After Upgrading

    • Test new slash commands: Run /speckit.constitution or another command to verify everything works
    • Review release notes: Check GitHub Releases for new features and breaking changes
    • Update workflows: If new commands were added, update your team's development workflows
    • Check documentation: Visit github.io/spec-kit for updated guides
